# 使用 Cursor 开发开源项目的修改流程
本文档梳理了使用 Cursor 编辑器修改开源项目的标准化流程，旨在帮助你规范开发步骤、提升效率。

## 一、前期准备与方案设计
### 1.1 明确核心目标
- 确定需要修改/新增的**具体功能点**（建议用清单形式列出，避免遗漏）
- 统一代码风格与项目既有形式（参考项目的 `.editorconfig`、`README` 或贡献指南）
- 明确修改的边界（不改动无关模块，降低回归风险）

### 1.2 生成 Cursor 配置文件
为了让 Cursor 更贴合项目开发需求，需创建/完善以下配置文件：

#### 1.2.1 `cursorignore`（忽略文件配置）
作用：告诉 Cursor 无需分析/处理的文件/目录，提升编辑器性能，避免干扰。
```ini
# cursorignore - 参考 .gitignore 格式
# 忽略依赖目录
node_modules/
vendor/
# 忽略构建产物
dist/
build/
# 忽略临时文件
*.log
*.tmp
# 忽略系统文件
.DS_Store
.idea/
```

#### 1.2.2 `cursorrule`（自定义规则配置）
作用：定义 Cursor 代码生成/补全的规则，匹配项目风格。
```ini
# cursorrule - Cursor 自定义规则
# 代码风格
indent_style = space
indent_size = 4
# 语言特性
javascript.enable_es6 = true
python.version = 3.9
# 生成偏好
generate_comments = true  # 强制生成注释
avoid_magic_numbers = true # 避免魔法值
match_project_naming = true # 匹配项目命名规范
```

### 1.3 输出修改设计方案
建议以 Markdown 或文档形式记录，内容至少包含：
- 功能修改的核心逻辑（流程图/伪代码）
- 涉及的核心文件/模块清单
- 预期的输入输出效果
- 潜在的风险点（如兼容性、性能影响）

## 二、编码实现
### 2.1 基础要求
- 基于设计方案编写代码，优先复用项目已有工具类/函数
- 遵循 Cursorrule 定义的风格，保持代码一致性
- 关键逻辑添加注释（说明设计思路、特殊处理原因）
- 小步提交（每完成一个子功能即提交代码，提交信息清晰）

### 2.2 Cursor 辅助技巧
- 使用 `Cmd/Ctrl + K` 调出指令面板，描述需求生成基础代码
- 选中代码块，通过 Cursor 对话功能快速优化/重构
- 利用 Cursor 的「解释代码」功能，验证对原有代码的理解是否准确

## 三、功能测试与细节调优
### 3.1 功能测试
- 编写单元测试（覆盖修改的核心逻辑）
- 手动测试关键场景（正向/反向用例）
- 验证边界条件（如空值、异常输入）

### 3.2 Bug 排查与风格调整
- 通过 Cursor 的「调试代码」功能定位问题
- 对照项目风格指南，调整代码格式、命名等细节
- 修复 Bug 后重新测试，确保无回归问题

## 四、打包脚本编写与项目发布
### 4.1 编写 Shell 打包脚本
示例脚本（可根据项目类型调整）：
```bash
#!/bin/bash
# 项目打包脚本 build.sh
set -e # 遇到错误立即退出

# 1. 检查依赖
echo "检查项目依赖..."
npm install # 前端项目示例，根据项目替换为 pip install/composer install 等

# 2. 执行构建
echo "开始构建项目..."
npm run build # 替换为项目实际构建命令

# 3. 打包产物
echo "打包构建产物..."
BUILD_DIR="dist"
ZIP_NAME="project-$(date +%Y%m%d%H%M%S).zip"
zip -r $ZIP_NAME $BUILD_DIR

# 4. 输出结果
echo "打包完成！产物路径：$(pwd)/$ZIP_NAME"
```

### 4.2 脚本使用说明
1. 赋予脚本执行权限：`chmod +x build.sh`
2. 运行脚本：`./build.sh`
3. 验证打包产物完整性（解压后测试功能）

---

### 总结
1. 前期准备核心是**明确目标+配置Cursor**，确保开发方向不偏离、编辑器适配项目；
2. 编码与测试阶段需**小步迭代+全面验证**，降低 Bug 率和返工成本；
3. 打包环节通过**标准化脚本**提升效率，保证产物一致性。
